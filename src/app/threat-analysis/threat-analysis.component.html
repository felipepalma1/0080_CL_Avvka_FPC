STRIDE Threat Model

STRIDE is a mnemonic to help discover threats in software and systems. STRIDE stands for Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, and Elevation of Privilege.

Each element of STRIDE helps the threat modeler to focus on potential threats to the entity being modelled to help create more complete threat models and attempt to reduce the likelihood of missing threats. It is possible that there will be no threats discovered for any element of STRIDE, but every element should be considered. For example a given flow may not have Tamper related threat because the transferred by the flow is encrypted with Barricade, and the connection the flow uses is TLS 1.2 which provides encryption for the flow

  • Spoofing: Pretending to be something or someone you're not
  • Tampering: Modifying something you're not supposed to modify. It can include packets on the wire or wireless, data on disk, or data in memory
  • Repudiation: Claiming you didn't do something (regardless of whether you did or not)
  • Information Disclosure: Exposing information to people wjo are not authorized to see it
  • Denial of Service: Attacks designed to prevent a system from providing service, including by crashing it, making it unusably slow or filling all its storage
  • Elevation of Privilege: A program or user is able to perform actions that theydo not have authority to perform

The threat modeling process is an iterative process that begins with the system conceptualization and high level design (Business Case level), iterating through the lower level design, and then implementation. Domain-Driven Design Concepts and Terms

Problem Domain

The specific problem the software you are working on is trying to solve

Core Domain

The key differentiator for the customer business; something they must do well and cannot outsource

Sub-Domains

Separate applications or features for which our software must support or interact

Bounded Context

A specific responsability with explicit boundaries that separate it from other parts of the system

Context Mapping

The process of identifying bounded context and their relationships to one another

Shared Kernel

Part of the model shared by two or more teams, who agree not to change it without collaboration

Ubiquitous Language

A language using terms from the domain model that programmers and domain experts use to discuss the system
